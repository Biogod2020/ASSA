# Design Specification: ASSA V3.2 (Autonomous Self-Sovereign Agent)
**Date**: 2025-05-22
**Status**: Finalized Blueprint
**Version**: 3.2 (Deep Memory Integration, Immutable Ledger, & L3 Promotion)

## 1. Architectural Vision & Core Principles

ASSA V3.2 transforms the Gemini CLI into a self-evolving, intelligent entity using a strictly decoupled L1/L2/L3 memory architecture. It guarantees **Zero Latency** in the main chat thread, prevents context bloat, and utilizes an **Immutable Ledger** to learn from both successes and user-initiated rewinds. Crucially, it features an autonomous promotion pipeline that elevates mature local project wisdom into a global, cross-project knowledge base.

### The Three Ironclad Rules:
1. **Zero Latency (Non-Blocking)**: The main chat thread must never wait for an LLM to perform background knowledge distillation or global synchronization.
2. **Anti-Context Bloat (Semantic Compression)**: Raw conversation histories and massive Git diffs are strictly prohibited from entering the active context. Only highly compressed, distilled patterns are allowed.
3. **Immutable Ledger (防时光倒流)**: No memory is ever physically deleted. All interactions, including user rewinds, are logged in an append-only ledger via a rigorous State Machine to provide both positive and negative training data.

## 2. Deep Memory Architecture: The File Ecosystem

The memory system is structured across three logical tiers: L1 (Transient Cache), L2 (Local Project Storage), and L3 (Global Universal Knowledge).

### A. L1: The Immutable Ledger (The Core State Machine)
**File**: `./.memory/evolution_ledger.json`

This is the heart of the system's immediate awareness. It is an append-only array of JSON objects tracking every realization the Agent makes in the current session.

**Schema**:
```json
[
  {
    "session_id": "uuid-string",
    "message_id": "uuid-string",
    "timestamp": "ISO-8601-string",
    "status": "PENDING | PROCESSED | REWOUND",
    "type": "positive_reinforcement | negative_correction | explicit_instruction",
    "payload": {
      "rule": "Actionable string (e.g., 'Enforce let/const, avoid var')",
      "context": "Brief explanation of why this was triggered",
      "tags": ["frontend", "refactor", "style"]
    },
    "git_anchor": "commit-hash-if-available"
  }
]
```

**State Machine Definition**:
- **`PENDING` (待处理)**: The active L1 Cache. The Main Agent just generated this signal. It is immediately injected into the next prompt for zero-latency awareness, but it hasn't been deeply analyzed yet.
- **`PROCESSED` (已归档)**: The Distiller Sub-agent has consumed this record, analyzed it alongside Git diffs, and written a generalized rule to the L2 `patterns.md`. It is removed from the L1 cache.
- **`REWOUND` (已撤销)**: The user executed a `/rewind` that erased the `message_id` associated with this record. The Distiller Sub-agent will now use this as **negative training data** (learning what *not* to do).

### B. L2: Local Project Wisdom
**Location**: `./.memory/`

This layer overrides or supplements the Global Layer for the specific repository. It represents the "mature" understanding of the current codebase.

1. **`patterns.md`**: The L2 Storage. This is where the Distiller Sub-agent writes its finalized, project-specific architectural rules extracted from the `evolution_ledger.json`.
2. **`decisions.md`**: A log of major technical trade-offs made specifically for this codebase to prevent the Agent from second-guessing past choices.
3. **`local_habits.md`**: Project-specific stylistic choices that differ from the user's global defaults.

### C. L3: Global Layer (The Soul & Universal Knowledge)
**Location**: `~/.gemini/assa/`

This layer applies to every project the user works on. It defines "Who the Agent is" and the universal truths it has learned across all projects.

1. **`SOUL.md`**: The unbreakable core persona. Contains the primary directives, the definition of the `submit_memory_signal` tool, and baseline behavioral constraints.
2. **`USER_HANDBOOK.md`**: The accumulated profile of the user's implicit habits (e.g., "User prefers 4-space indentation," "User hates boilerplate comments").
3. **`LIBRARY/`**: A directory of highly specialized, domain-specific markdown files (e.g., `PYTHON_SOTA.md`, `REACT_ARCHITECTURE.md`). These are massive and **never** injected directly into the chat.
4. **`LIBRARY/index.json`**: The Semantic Map. A highly compressed directory mapping tags to `LIBRARY/` files. The Context Router injects this index so the Agent knows *how* to look up deep knowledge if needed.
5. **`ANTI_PATTERNS.md`**: The Wall of Shame. Global anti-patterns learned from `REWOUND` states across multiple projects.

### D. Memory Resolution Hierarchy
When the Context Router builds the prompt, the priority is strictly enforced:
1. `SOUL.md` (Absolute Authority)
2. `PENDING` Ledger Items (Immediate Context / L1)
3. `./.memory/patterns.md` (Project Specific L2)
4. `~/.gemini/assa/USER_HANDBOOK.md` (Global Habits / L3)
5. `LIBRARY/index.json` (Awareness of deep knowledge / L3)

## 3. Data Flow & Execution Mechanics

### A. L1: Agentic Information Compression (Main Thread)
We rely on the LLM's own intelligence to compress context, rather than brittle Python regex.

- **Tool Registration**: The Main Agent is provided with a native tool:
  `submit_memory_signal(type: string, rule: string, context: string, tags: string[])`
- **Behavior**: When the Agent detects the user correcting its code, or praising a specific approach, it autonomously calls this tool.
- **Action**: The payload is instantly appended to `./.memory/evolution_ledger.json` with the status `PENDING`.

### B. L2: The Middleware Interceptor (The Hooks)
A single, highly cohesive Python script (`assa_middleware.py`) manages all traffic.

**1. `BeforeAgent` Hook (State Alignment & Hot-Reloading)**:
- **Time-Travel Defense**: Reads the incoming `transcript`. Iterates through the ledger. If a `message_id` in the ledger no longer exists in the transcript, it forcibly cascades its status to `REWOUND`.
- **Context Injection**: Reads all `PENDING` ledger items, `patterns.md`, and the `index.json`, concatenating them into the `additionalContext` field.

**2. `AfterTool` Hook (Behavioral Anchors)**:
- **Git Commit Interception**: If `tool_name == "run_shell_command"` and the command is a `git commit`, the hook registers a "Positive Anchor" in the ledger.
- **Trigger**: The commit action serves as a definitive signal that the current codebase state is approved. It then triggers the Distiller Sub-agent.

### C. The Distiller Sub-Agent (L1 -> L2)
The `distiller.md` Sub-agent handles the conversion of immediate feedback into project rules.

- **Spawning**: Triggered by a `git commit` OR when `PENDING` records > 5. Spawned via a non-blocking `subprocess` (`gemini --agent distiller`).
- **Data Ingestion**: The middleware pipes the `PENDING` and `REWOUND` records, along with the recent `git diff`, into the Sub-agent's standard input.
- **Processing**: The Distiller abstracts the raw rules into permanent architectural guidelines for the project.
- **Commit**: The Distiller rewrites `./.memory/patterns.md`.
- **Ledger Update**: The middleware marks the processed ledger items as `PROCESSED`.

## 4. The Global Syncer: L2 -> L3 Promotion Mechanism

The most advanced feature of ASSA V3.2 is its ability to recognize when a local project pattern is universal enough to become global knowledge. This is handled by a specialized **Syncer Sub-Agent** (`syncer.md`).

### A. Triggers for Global Synchronization
The Syncer Sub-agent is spawned asynchronously under the following conditions:
1. **Implicit Signal (`git push`)**: Intercepted via the `AfterTool` hook. Pushing code indicates a major milestone and a stable state suitable for cross-project abstraction.
2. **Explicit Command (`/assa promote`)**: A custom CLI command allowing the user to manually force the agent to review recent local patterns for global promotion.
3. **Dialogue Heuristics**: If the Main Agent detects strong, generalized statements from the user (e.g., "I always want you to use this structure from now on"), it calls a secondary tool: `request_global_promotion(topic: string)`.

### B. The Syncer Workflow
When triggered, the `syncer.md` Sub-agent performs the following in a background `subprocess`:
1. **Audit**: Reads the local `./.memory/patterns.md` and `./.memory/decisions.md`.
2. **Cross-Reference**: Compares local patterns against the existing global `~/.gemini/assa/LIBRARY/` files to identify overlaps, conflicts, or entirely new paradigms.
3. **Abstraction**: Rewrites highly specific local code rules into generalized, framework-agnostic principles.
4. **Injection**: 
   - Appends or modifies the relevant files in the global `LIBRARY/`.
   - Updates the global `LIBRARY/index.json` so the Context Router can map the new knowledge.
5. **Local Cleanup**: Removes the promoted patterns from the local `./.memory/patterns.md` to prevent redundancy and keep the L2 cache lean.

## 5. Engineering Implementation Tasks

1. **Extension Scaffolding (`gemini-extension.json`)**:
   - Define the `BeforeAgent` and `AfterTool` hooks.
   - Define the `submit_memory_signal` and `request_global_promotion` local tools.
2. **Ledger Management (`ledger_utils.py`)**:
   - Create robust JSON read/write handlers with atomic locks to prevent race conditions.
3. **Middleware Logic (`assa_middleware.py`)**:
   - Implement the Transcript vs. Ledger comparison to detect rewinds.
   - Implement the Git command interceptors for `commit` (Distiller trigger) and `push` (Syncer trigger).
4. **Sub-Agent Profiles (`agents/`)**:
   - `distiller.md`: Prompt for converting L1 `PENDING` records into L2 `patterns.md`.
   - `syncer.md`: Prompt for auditing L2 patterns and promoting them to the L3 `LIBRARY/` and updating the `index.json`.
